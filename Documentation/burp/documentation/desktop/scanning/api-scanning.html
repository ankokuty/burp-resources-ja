Burp Scannerを使用したAPIのスキャン

Burp Scannerは、JSONベースとYAMLベースの両方のAPI定義をスキャンして脆弱性を検出できます。デフォルトでは、クローラが検出したAPI定義を解析して、サポートされているメソッドとパラメーターとともに、潜在的なエンドポイントを識別しようとします。スキャンを開始するときに、API定義のURLを明示的に指定もできます。検出したエンドポイントに基づいて、Burp Scannerは新たにクロールおよび診断する場所を取得できます。

アプリケーションの他の部分をスキャンする場合と同様に、クロールフェーズで使用されたものと同じ一連のリクエストがAPIエンドポイントの診断にも使用されます。

注

必要に応じて、スキャン設定で"API定義の解析"クロールオプションの選択を解除すると、APIスキャンを無効にできます。このオプションは"その他"にあります。

APIスキャンの前提条件

Burp Scannerは、API定義をスキャンするためにそれを解析できる必要があります。現在、これは次の要件を満たすAPI定義でのみ可能です:

  • API定義は、OpenAPIバージョン3.xx仕様である必要があります。クローラは、検出されたドキュメントがAPI定義であるかどうかを識別するために、定義の最上位のキーにopenapiを使用します。
  • API定義には、外部参照を含めることはできません。

これらの要件を満たさない定義は、スキャン中にスキップされます。

API定義から場所を取得する

クローラがAPI定義を解析するとき、エンドポイントごとに複数の場所を作成することがよくあります。エンドポイントが複数のメソッドをサポートしている場合、それぞれに個別の場所が作成されます。同様に、スコープ内に複数のAPIサーバがある場合、単一のメソッドとエンドポイントの組み合わせは、それぞれの異なるサーバへの同じ呼び出しを表す別々の場所が生成されます。

例えば、特定のエンドポイントがGETとPUTメソッド両方をサポートするとします。スコープ内にサーバが1つしかない場合、このエンドポイントから2つの場所が派生します。ただし、サーバが3つある場合は、合計6つの新しい場所になります。

APIスキャン中のパラメータ処理

クロール中、各エンドポイントのパラメータが定義される方法は、Burp Scannerが送信するリクエストの数に影響します:

  • オプションのパラメータが定義されている場合、クローラは少なくとも2つのリクエストをそのエンドポイントに送信します。1つは必須パラメーターのみを含むリクエストで、もう1つはすべてのオプションパラメータも含むリクエストです。
  • 列挙型の場合、クローラはパラメータに許可された値ごとに個別のリクエストを送信します。

これにより、各エンドポイントを最大限にカバーできます。

各リクエストのパラメータに使用される値も、API定義によって部分的に決定されます。パラメータにサンプル値が指定されている場合、クローラはそれぞれの場合に最後のサンプルを使用します。それ以外の場合は、適切なカスタム値が生成されます。

APIスキャンの制限

クローラがAPI定義を解析しようとする場合、次の制限が適用されます。エンドポイントがこれらの制限に準拠していない場合、スキャンから除外されます。いずれの場合も、イベントログをチェックして、特定のエンドポイントがスキップされた理由を確認できます。

  • スキャンでは引き続きBurpの通常の認証処理機能を使用しますが、クローラは現在、エンドポイントレベルで実装されている認証を処理できません。
  • サーバパラメータとパスパラメータは、それらが列挙型である場合、または定義で値の例が提供されている場合にのみサポートされます。
  • 次のいずれかがリクエストに存在する必要があるエンドポイントはサポートされていません:
    • 追加のHTTPヘッダー
    • arrayタイプのクエリまたはボディパラメータ。ただし、このタイプのJSONボディパラメーターはサポートされていることに注意してください。
    • XMLタイプのクエリまたはボディパラメータ
    • タイプが混合しているクエリまたはボディパラメータ。例えば、application/x-www-form-urlencoded ボディ内のJSONパラメータ、など。
    • JSONタイプのクエリパラメータ。ただし、このタイプのボディパラメータはサポートされていることに注意してください。
    • multipartタイプのボディパラメータ
